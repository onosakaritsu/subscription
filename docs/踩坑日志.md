# 踩坑日志

## 2026-07-02｜架构迁移｜无 Apple Development Team 时改为 Web App

**问题现象：**  
项目原方案依赖 SwiftUI / WidgetKit / App Group。用户没有 Apple Development Team，真实 App Group 和 macOS Widget 签名闭环无法完成。

**原因分析：**  
WidgetKit + App Group 的共享容器验收不仅需要源码配置，还需要 Apple Developer 侧的 Bundle ID、App Group capability 和有效 provisioning profile。没有 Team 时，继续推进原生 Widget 会长期卡在环境条件上。

**解决方案：**  
项目迁移为本地优先 Web App：Node.js 原生 HTTP 服务 + 静态前端 SPA + 本地 JSON 文件。该方案不需要证书、签名、App Group，也不需要下载第三方依赖。

**复盘经验：**  
当核心平台能力被账号、证书或审核链路卡住时，应优先把产品目标拆出来，选择能闭环交付的技术形态。订阅管理本身不依赖原生 Widget，Web App 更适合当前约束。

## 2026-07-02｜产品范围｜网页形态取消小组件

**问题现象：**  
迁移到 Web 后，曾考虑用一个 Widget 风格页面替代桌面 Widget。用户明确指出既然是网页，就不需要小组件。

**原因分析：**  
桌面 Widget 是原生 macOS 方案下的入口形态；迁移到网页后，继续模拟小组件会增加无关页面和 API，偏离主任务。

**解决方案：**  
删除 `/api/widget` 设计，不创建独立 Widget 页面。Web V1 聚焦订阅管理主界面、统计、筛选、导入导出和本地持久化。

**复盘经验：**  
技术迁移时不要机械保留旧形态的所有功能。需要重新判断哪些是核心用户价值，哪些只是旧平台的实现方式。

用户要求严格在 `~/Documents/Workspace/Project_03_subscription-widget` 中继续开发，但初次查找时该目录不存在。

**原因分析：**

**解决方案：**

用户要求“不得移动或重命名项目目录”时，如果目录不存在，必须先确认是否允许新建，不能自行推断或把其他目录改名。

## 2026-07-02｜XcodeGen｜App target 缺少 Info.plist 导致嵌入 Widget 校验失败

**问题现象：**  
无签名构建时 Swift 源码已编译通过，但 `ValidateEmbeddedBinary` 阶段提示找不到 `SubscriptionManager.app/Contents/Info.plist`。

**原因分析：**  
初版 `project.yml` 只给 Widget Extension 配置了 `Info.plist`，App target 依赖默认生成行为不够明确，导致打包校验阶段缺少主 App 的 plist。

**解决方案：**  
新增 `SubscriptionManagerApp/Info.plist`，并在 `project.yml` 中为 `SubscriptionManager` target 显式声明 `info.path` 和基础属性。重新运行 XcodeGen 后构建通过。

**复盘经验：**  
XcodeGen 工程应把 App 和 Extension 的 Info.plist 都写进源配置，尤其是嵌入 extension 的 macOS App，避免校验阶段才暴露打包文件缺失。

## 2026-07-02｜Xcode Preview 宏｜受限环境中 Preview 宏阻断命令行构建

**问题现象：**  
包含 `#Preview` 的 Widget 文件在命令行构建时失败，提示外部宏实现 `PreviewsMacros.Common` 找不到，`swift-plugin-server` 返回异常。

**原因分析：**  
当前执行环境对 Xcode 插件/宏服务存在限制；V1 不依赖 Preview 宏，保留它会让命令行构建稳定性变差。

**解决方案：**  
移除 App 和 Widget 中的 `#Preview` 块，改用实际构建和测试验证。

**复盘经验：**  
对 macOS/WidgetKit 项目，Preview 宏属于开发便利项，不应阻塞 CI 或命令行验证；在受限环境优先保证 `xcodebuild` 稳定。

## 2026-07-02｜Widget 布局｜避免标题上方空白和新增后行上移

**问题现象：**  
需求要求修复此前 Widget 标题上方空白过大，以及添加项目后列内容整体上移的问题。

**原因分析：**  
Widget 内容如果依赖自适应 spacer 或不稳定列表高度，数据项数量变化后容易出现垂直位置漂移。

**解决方案：**  
Widget 顶层使用 `frame(maxWidth:maxHeight:alignment: .topLeading)`，header 使用固定高度，行组件使用固定高度；Small、Medium、Large 分别限制展示数量。

**复盘经验：**  
Widget 信息密度高，优先使用稳定的 `VStack` / `HStack` / fixed frame；不要让数据条数变化改变标题或列的基准位置。

## 2026-07-02｜App Group｜本地 Sign to Run Locally 不等于真实共享容器验收

**问题现象：**  
工程中 App 和 Widget 均声明了 App Group entitlements，正常本地签名构建成功，但用 `codesign -d --entitlements :-` 检查最终产物时，只看到 `com.apple.security.get-task-allow`，未看到 `com.apple.security.application-groups`。

**原因分析：**  
当前使用 “Sign to Run Locally”，没有配置真实 Development Team 和包含 App Group capability 的 provisioning profile。构建成功、代码中使用 Group ID、entitlements 源文件存在，并不等于最终签名产物已获得 App Group 权限。

**解决方案：**  
保留 App Group 源配置和共享存储代码；存储层在 App Group 容器不可用时降级到 Application Support，便于继续开发。后续必须配置真实 Team/profile 后复验签名产物 entitlement、主 App 写入、Widget 读取和重启持久化。

**复盘经验：**  
App Group 验收分三层：代码完成、构建通过、真实共享可用。最终必须以签名产物 entitlement 和实际容器读写为准。

## 2026-07-02｜XcodeGen｜entitlements 必须写在 project.yml 源配置

**问题现象：**  
审阅时发现 `Config/SubscriptionManager.entitlements` 和 `Config/SubscriptionRenewalWidget.entitlements` 是空 `dict`，与 README 中“已声明 App Group”的说法不一致。直接手工修改 entitlements 文件后，再运行 `xcodegen generate` 又会被覆盖为空。

**原因分析：**  
XcodeGen 会根据 `project.yml` 重新生成 entitlements 文件；如果只写 `entitlements.path`，没有写 `entitlements.properties`，源文件会被生成器重置。

**解决方案：**  
在 `project.yml` 的 App 和 Widget target 下增加 `entitlements.properties.com.apple.security.application-groups`，值为 `$(APP_GROUP_IDENTIFIER)`。重新生成工程后，两个 entitlements 源文件均包含 App Group 声明。

**复盘经验：**  
XcodeGen 项目里 `.xcodeproj`、Info.plist、entitlements 都应以 `project.yml` 为事实来源；不能只改生成物或会被生成器覆盖的文件。

## 2026-07-02｜管理列表｜停用订阅不能从侧栏消失

**问题现象：**  
侧栏列表复用了 Widget 的 `upcomingRenewals` 逻辑，而该逻辑会过滤掉 `isEnabled == false` 的订阅。用户把订阅停用后，该项目会从管理列表消失，后续无法方便地重新编辑或启用。

**原因分析：**  
“即将续费展示”和“订阅管理列表”是两个不同视图语义：Widget 和汇总区应只展示启用项目，主 App 管理列表应展示全部项目。

**解决方案：**  
新增 `sortedForManagement()` / `managementItems()`，侧栏使用管理排序但不过滤停用项目；Widget 仍使用 `upcomingRenewals()` 过滤启用项目。补充单元测试覆盖停用项仍可见。

**复盘经验：**  
UI 复用排序逻辑可以，但不要复用过滤语义；管理界面应优先保证对象可找回、可编辑。

## 2026-07-02｜xcodebuild test｜沙箱内无法连接 testmanagerd

**问题现象：**  
沙箱内运行 `xcodebuild test` 时测试包已构建，但测试启动阶段失败，提示无法与 `com.apple.testmanagerd.control` 通信。

**原因分析：**  
macOS 测试运行需要访问本机测试管理服务，当前沙箱限制了该 XPC 通信。

**解决方案：**  
按权限规则请求非沙箱执行同一条 `xcodebuild test` 命令。非沙箱运行后 2 个单元测试全部通过。

**复盘经验：**  
区分“测试代码失败”和“测试运行器被环境拦截”。遇到 `testmanagerd` 通信错误时，应在同一命令下用合规权限复跑。

## 2026-07-02｜SwiftUI 状态｜编辑器草稿需要跟随选中项重置

**问题现象：**  
订阅编辑器使用 `@State` 保存当前草稿。若用户从左侧列表切换到另一条订阅，SwiftUI 可能复用编辑器视图实例，使草稿继续停留在上一条记录。

**原因分析：**  
`@State` 的初始值只在视图状态首次创建时生效；父视图传入的新 `item` 不会自动覆盖内部草稿。

**解决方案：**  
在 `SubscriptionEditorView` 中保留传入的 `item`，并通过 `.onChange(of: item)` 同步重置 `draft` 与 `amountText`。

**复盘经验：**  
SwiftUI 编辑表单如果使用本地草稿状态，必须明确处理外部选中对象变化；否则列表选择和表单内容可能脱节。

## 2026-07-02｜Phase 2｜状态、排序和统计必须放在领域层

**问题现象：**  
Phase 2 需要续费状态标签、列表排序、近期续费摘要和统计卡片。如果只在前端 DOM 渲染里临时判断，会导致 API、测试和后续功能复用困难。

**原因分析：**  
续费状态、排序和金额折算都属于订阅领域规则，不属于纯展示逻辑。前端只负责展示，后端也需要返回一致的派生结果。

**解决方案：**  
将 `renewalStatus`、`sortForManagement`、`summarizeSubscriptions`、`monthlyEquivalent`、`yearlyEquivalent` 等逻辑统一放入 `src/domain/subscriptions.mjs`，并补充 Node.js 测试。

**复盘经验：**  
涉及业务规则的 UI 标签和排序都应先领域化，再渲染；否则后续导出、提醒、统计会出现多套不一致逻辑。

## 2026-07-02｜数据安全｜保存成功后自动备份但不阻断主流程

**问题现象：**  
本地 JSON 文件容易被误删或损坏，需要自动备份。但备份失败如果阻断主保存，会让用户保存订阅失败。

**原因分析：**  
主数据写入和备份写入的优先级不同：主保存必须可靠，备份是保护层。备份目录权限、磁盘空间或文件系统错误都可能失败。

**解决方案：**  
主数据先原子写入 `data/subscriptions.json`，成功后尝试写入 `data/backups/subscriptions-backup-YYYY-MM-DD-HH-mm-ss.json`。备份失败只输出服务端日志，不影响主保存。最多保留 20 份备份。

**复盘经验：**  
本地优先应用必须保护用户数据，但保护逻辑不能反过来破坏主要保存路径。

## 2026-07-02｜GitHub 默认展示｜README.md 应使用中文

**问题现象：**  
项目面向中文使用场景，但 GitHub 默认展示 `README.md`。如果默认 README 是英文，用户打开仓库时信息不够直接。

**原因分析：**  
GitHub 只会默认渲染根目录 `README.md`。多语言文档应以主要用户语言作为默认文件，再提供其他语言链接。

将 `README.md` 改为中文版，新增 `README.en.md` 保留英文文档入口。

**复盘经验：**  
开源项目的默认 README 应服务主要读者；多语言不是平均展示，而是默认路径清晰、辅助语言可达。

## 2026-07-05｜Phase 3｜恢复备份前必须先保护当前数据

**问题现象：**
备份恢复功能会覆盖 `data/subscriptions.json`。如果目标备份损坏、选错文件或恢复中途失败，可能造成当前数据丢失。

**原因分析：**
恢复不是普通读取操作，而是高风险写入操作。它必须先校验目标备份，并在覆盖前保存当前状态。

**解决方案：**
新增安全文件名校验，禁止路径穿越；恢复前先读取并校验目标备份 JSON 和订阅结构，再创建 `subscriptions-before-restore-YYYY-MM-DD-HH-mm-ss.json`，最后写入主数据。损坏备份不会执行恢复。

**复盘经验：**
本地优先应用的数据恢复功能必须把“可恢复当前状态”作为第一原则。恢复闭环不是只有恢复入口，还包括恢复前保护、失败不破坏、错误可理解。

## 2026-07-05｜Phase 3｜导入 JSON 也属于高风险覆盖操作

**问题现象：**
JSON 导入会整体替换当前订阅数据。如果导入文件格式错误或结构不兼容，不能让当前数据被半覆盖。

**原因分析：**
导入和恢复一样，本质都是覆盖主数据。只做 JSON.parse 不足以保证业务结构可被当前系统识别。

**解决方案：**
导入前先校验 JSON 格式和订阅数组结构；校验通过后创建 `subscriptions-before-import-YYYY-MM-DD-HH-mm-ss.json`，再写入主数据。导入失败时当前数据保持不变。

**复盘经验：**
所有覆盖式数据入口都应该统一遵守“先校验、再备份、后覆盖”的顺序。

## 2026-07-05｜Phase 4｜状态不能只靠颜色表达

**问题现象：**
Phase 4 需要强化订阅状态识别。如果只依赖颜色，深色模式和可访问性都会变差，且不同区域可能出现颜色含义不一致。

**原因分析：**
订阅列表、最近续费摘要和续费日历都在展示同一套续费状态。状态语义应先统一，再映射到颜色和标签。

**解决方案：**
领域层增加状态 metadata，前端统一使用状态 badge 渲染函数和 CSS 状态变量。每个状态都有文字标签、tone、priority 和 description。深色模式下单独调整状态色变量，避免低对比度。

**复盘经验：**
视觉状态体系应把“语义一致”放在“颜色好看”之前。颜色只是辅助，文字标签和一致排序才是长期可维护的状态表达。

## 2026-07-05｜Phase 4｜覆盖式操作必须保留恢复路径

**问题现象：**
外部备份恢复、导入 JSON、内部备份恢复都会覆盖当前主数据。

**原因分析：**
这些操作属于高风险写入，失败或误操作都可能影响用户真实数据。

**解决方案：**
所有覆盖式入口继续遵循“先校验、再备份、后覆盖”。外部恢复不使用 multipart 上传，而是由前端解析 JSON 后通过 body 传给后端，后端再次校验结构。

**复盘经验：**
本地工具的安全感来自可回退。功能越接近日常使用，越要减少不可逆操作。

## 2026-07-05｜Phase 4｜Apache 2.0 正文不能混入额外免责声明

**问题现象：**
项目需要从 MIT 调整为 Apache License 2.0，同时补充更明确的风险提示和二创责任边界。

**原因分析：**
Apache License 2.0 是标准开源许可证。若直接在 LICENSE 正文里加入中文解释、额外免责或限制条款，可能影响 GitHub 识别，也可能与标准授权边界冲突。

**解决方案：**
根目录 `LICENSE` 使用 Apache License 2.0 标准全文，`package.json` 使用 SPDX 标准值 `Apache-2.0`。额外风险提示放入 `DISCLAIMER.md`、README 和相关文档，明确其属于项目说明，不改写许可证正文。

**复盘经验：**
许可证正文和项目风险提示应分层管理。标准许可证负责授权，免责声明负责解释风险边界，二者不能混成自定义许可证。


## 2026-07-05｜Phase 5｜完整性检查必须只读

**问题现象：**
本地 JSON 数据可能出现空名称、负金额、非法日期、重复 id 或未知币种。若检查功能顺手自动修复，可能误改用户真实数据。

**原因分析：**
完整性检查的目标是让用户知道风险，不是替用户做不可逆修改。自动修复应在后续阶段单独设计确认流程。

**解决方案：**
新增 `GET /api/integrity` 和 `diagnoseSubscriptions()`，只返回 error / warning，不写入 `data/subscriptions.json`。未知币种作为 warning 保留原代码，避免破坏已有数据。

**复盘经验：**
诊断和修复应分层。尤其是本地优先工具，任何会修改真实数据的行为都必须显式确认并可回退。

## 2026-07-05｜Phase 5｜截图验收工具不能进入项目依赖

**问题现象：**
Phase 5 允许真实浏览器截图验收，但项目仍要求零第三方运行时依赖。

**原因分析：**
Playwright、系统浏览器或其他截图工具只属于本地验收手段，不应写入 `package.json` 或提交缓存、node_modules、临时截图产物。

**解决方案：**
截图验收只作为临时本地流程。若保留截图，放在 `docs/screenshots/phase5/` 并使用演示数据；不提交真实个人订阅数据和工具缓存。

**复盘经验：**
验收工具和产品依赖要严格区分。工具可以帮助看见问题，但不能改变项目技术约束。


## 2026-07-06｜v1.0.0｜导入/恢复预览不能替代真正备份

**问题现象：**
正式版需要降低误导入、误恢复风险。仅靠浏览器确认弹窗不够，用户需要先看到文件内容和差异摘要。

**原因分析：**
导入和恢复都是覆盖式操作。预览可以帮助用户判断“是不是选对了文件”，但预览本身不提供回滚能力。

**解决方案：**
新增 `/api/import/preview` 和 `/api/backups/restore-uploaded/preview`，只读返回订阅数量、币种分布、启用/停用数量、新增/覆盖/缺失估算和前几条订阅。真正导入或恢复时仍由后端执行“先校验、再备份、后覆盖”。

**复盘经验：**
预览解决认知风险，备份解决恢复风险。两者不能互相替代，发布版必须同时具备。

## 2026-07-06｜v1.0.0｜公开仓库必须默认不含真实数据

**问题现象：**
本地优先工具的数据文件就在项目目录中，发布前如果不审计 `.gitignore` 和 tracked files，容易把真实订阅、备份或截图提交到公开仓库。

**原因分析：**
`data/subscriptions.json`、`data/backups/*.json` 和浏览器截图都可能包含私人订阅信息。即使功能稳定，仓库不干净也不能作为正式可用版发布。

**解决方案：**
v1.0.0 发布清单明确检查真实数据、备份 JSON、node_modules、日志、缓存、截图隐私和许可证文件。截图验收使用演示数据，正式截图放入 `docs/screenshots/v1.0.0/`。

**复盘经验：**
本地优先项目的发布质量不只看功能，也要看仓库边界。默认安全的开源仓库会让用户更放心地 fork、审计和二次开发。
